Layer 1: Training Entry Point

Layer 2: Process-Level Parallelization
  File: src/parallel/process_coordinator.py
  Purpose: Use ProcessPoolExecutor to train on multiple CVEs in parallel
  Logic: 
    - Each CVE gets its own process
    - Each process calls skyrl_wrapper for that CVE
    - ProcessPoolExecutor manages process pool
    - Manages multiple progress bars (one per CVE)
  Note: Do NOT mix threading with Ray - Ray is handled internally by SkyRL

Layer 2.5: Progress Monitoring
  File: src/parallel/progress_monitor.py
  Purpose: Display real-time progress for parallel training
  Library: Use `tqdm` for terminal progress bars
  
  Requirements:
    - Multiple progress bars displayed simultaneously (one per CVE/task)
    - Each progress bar shows: [CVE-ID] Episode X/Y | Step Z/max_steps (XX%)
    - Progress bars update in real-time from different processes
    - Clean display without terminal corruption
  
  Implementation approach:
    - Use tqdm.tqdm() with position parameter for multiple bars
    - Each process gets its own progress bar at fixed position
    - Progress bar format: "[CVE-2024-XXXX] Ep 5/100 | Step 15/30 (50%)"
    - Update via shared state (multiprocessing.Manager or Queue)
  
  Example display during training:
    ```
    Training 4 CVEs in parallel...
    
    [CVE-2024-2624    ] Ep 12/100 | Step 23/30 (76%) |████████████░░░░| 76/100
    [CVE-2024-2771    ] Ep  8/100 | Step 10/30 (33%) |█████░░░░░░░░░░░| 33/100
    [CVE-2024-3094    ] Ep 15/100 | Step 28/30 (93%) |██████████████░░| 93/100
    [jenkins/CVE-18861] Ep  3/100 | Step  5/30 (17%) |███░░░░░░░░░░░░░| 17/100
    
    Overall: 4 tasks running | Avg episode: 9.5/100 | Avg step progress: 54.8%
    Elapsed: 00:15:32 | Est. remaining: 02:14:28
    ```
  
  Progress bar behavior:
    - Each bar updates independently as its process reports progress
    - Bars remain in fixed positions (no jumping around)
    - Step progress resets to 0% at start of each new episode
    - Episode counter increments when episode completes
    - Overall stats calculated from all active tasks
  
  Key functions:
    - create_progress_bars(task_ids) -> dict[task_id, tqdm_bar]
    - update_progress(task_id, episode, step, max_steps)
    - close_progress_bars()
    - format_bar_description(task_id, episode, step, max_steps) -> str

Layer 5: Gymnasium Environment Interface
  File: src/env/security_env.py
  Purpose: Provide Gymnasium-compliant environment for SkyRL
  Interface:
    - reset() -> (observation, info)
    - step(action) -> (observation, reward, terminated, truncated, info)
    - close()
  Responsibilities:
    - Select appropriate adapter based on task_type
    - Orchestrate reward computation
    - Standardize observations and actions
    - Report progress to progress_monitor (if enabled)
  Dependencies:
    - Creates adapter via EnvRegistry
    - Creates reward via RewardRouter
    - Updates progress dict (passed in config if parallel training)
  
  Progress integration:
    ```python
    class SecurityEnv(gym.Env):
        def __init__(self, config):
            self.task_id = config['task_id']
            self.progress_dict = config.get('progress_dict', None)
            self.current_episode = 0
            self.current_step = 0
            self.max_steps = config.get('max_steps', 30)
        
        def reset(self):
            self.current_step = 0
            self.current_episode += 1
            self._update_progress()
            # ... rest of reset logic
        
        def step(self, action):
            self.current_step += 1
            self._update_progress()
            # ... rest of step logic
        
        def _update_progress(self):
            if self.progress_dict is not None:
                self.progress_dict[self.task_id] = {
                    'episode': self.current_episode,
                    'step': self.current_step,
                    'max_steps': self.max_steps
                }
    ```

    - Docker Compose operations (docker-compose up/down)

    - Target container discovery
    - Command execution (_execute_bash, _execute_http)
    - All Docker operations handled internally
  
  Implementation details:
    - setup() : 
      ↳ Runs docker-compose to start target services
      ↳ Builds/starts attacker container internally
      ↳ Discovers containers and network
      ↳ Waits for services to be ready
    
    - teardown() : 
      ↳ Stops attacker container
      ↳ Runs docker-compose down for target services
      ↳ Cleans up resources
    
    - reset() : 
      ↳ Generates task description
      ↳ Returns initial observation with target info
    
    - step(action) : 
      ↳ Executes bash/http commands in attacker container
      ↳ Returns observation, reward, done, info
    
    - _build_attacker_image() :
      ↳ Inline Dockerfile definition
      ↳ Uses docker.from_env() to build image
      ↳ Task-specific tools installation
    
    - _execute_bash(command) :
      ↳ Uses container.exec_run() to execute in attacker
      ↳ Returns stdout/stderr
    
    - _execute_http(method, url, ...) :
      ↳ Uses container.exec_run() with curl/python
      ↳ Returns HTTP response
  
  Note: No abstraction layers (no docker_manager, no attacker_manager)
        Each adapter handles ALL Docker operations internally
        This is intentional for simplicity and stability

1. PARALLELIZATION STRATEGY
   - Use ProcessPoolExecutor for different CVEs (not threading!)
   - Let SkyRL handle Ray workers internally
   - Each process is independent, no shared state

2. NAMING CONVENTIONS
   - adapters (not adaptors)
   - docker (not dockers)
   - env (not main_framework)
   - Consistent snake_case for files

3. ADAPTER APPROACH (PRAGMATIC)
   - Copy adapters from infra/ and infra_v2/ UNCHANGED
   - No abstraction layers (no docker_manager, no attacker_manager)
   - Each adapter is self-contained with ALL Docker logic
   - Rationale: Working code, minimal risk, faster implementation
   - Trade-off: Some code duplication, but stable and tested

4. COMMAND EXECUTION
   - _execute_bash() and _execute_http() stay INSIDE adapters
   - Not extracted to util/ for simplicity
   - Each adapter has its own implementation
   - Keeps adapters fully self-contained

5. TESTING APPROACH
   - Python script, not bash
   - Reuses SecurityEnv from training
   - Works with ALL adapters (not just CVE-bench)

6. ENVIRONMENT INTERFACE
   - Keep security_env.py
   - No test_env.py yet (same env for train and test)
   - Gymnasium-compliant interface

5. PROGRESS BAR SYSTEM (IMPORTANT):
   
   Requirements:
   - Display multiple progress bars simultaneously (one per CVE/task)
   - Each bar shows: [Task-ID] Episode X/Y | Step Z/max_steps (percentage)
   - Real-time updates from different processes
   - Clean terminal display (no corruption)
   
   Implementation using tqdm:
   
   ```python
   from tqdm import tqdm
   from multiprocessing import Manager
   
   # In process_coordinator.py
   def start_parallel_training(task_ids):
       manager = Manager()
       progress_dict = manager.dict()
       
       # Create progress bars with fixed positions
       bars = {}
       for idx, task_id in enumerate(task_ids):
           bars[task_id] = tqdm(
               total=100,  # Percentage
               desc=f"[{task_id[:20]}]",
               position=idx,
               leave=True
           )
       
       # Start training processes
       with ProcessPoolExecutor() as executor:
           futures = {
               executor.submit(train_task, tid, progress_dict): tid 
               for tid in task_ids
           }
           
           # Monitor and update progress bars
           while futures:
               # Check progress_dict for updates
               for task_id, progress_info in progress_dict.items():
                   episode = progress_info['episode']
                   step = progress_info['step']
                   max_steps = progress_info['max_steps']
                   percentage = int((step / max_steps) * 100)
                   
                   # Update bar
                   bars[task_id].n = percentage
                   bars[task_id].set_description(
                       f"[{task_id[:20]}] Ep {episode} | Step {step}/{max_steps}"
                   )
                   bars[task_id].refresh()
   
   # In training function (called by each process)
   def train_task(task_id, progress_dict):
       for episode in range(num_episodes):
           for step in range(max_steps):
               # Training logic...
               
               # Update progress
               progress_dict[task_id] = {
                   'episode': episode,
                   'step': step,
                   'max_steps': max_steps
               }
   ```
   
   Display format:
   ```
   [CVE-2024-2624]     Ep 12/100 | Step 23/30 (76%) ████████████░░░░
   [CVE-2024-2771]     Ep 8/100  | Step 10/30 (33%) █████░░░░░░░░░░░
   [vulhub/jenkins]    Ep 15/100 | Step 28/30 (93%) ██████████████░░
   [xbow/challenge_1]  Ep 3/100  | Step 5/30  (17%) ███░░░░░░░░░░░░░
   ```
   
   Key considerations:
   - Use multiprocessing.Manager().dict() for shared progress state
   - Each progress bar has fixed position parameter
   - Update from main process, not worker processes (avoid conflicts)
   - Use tqdm.refresh() to update display
   - Close all bars properly on completion

PHILOSOPHY: PRAGMATIC OVER PERFECT
===============================================================================

infra_v4 takes a PRAGMATIC approach:

✅ NEW: Parallel execution, SkyRL integration, testing framework
📋 COPY: Adapters unchanged from infra/infra_v2 (proven, working code)
❌ SKIP: Abstraction layers that would require adapter refactoring

Why?
  1. Faster to implement (focus on new features)
  2. Lower risk (adapters are tested and stable)
  3. Easier to debug (all adapter logic in one place)
  4. Can always refactor later if needed

Key insight:
  The VALUE of infra_v4 is in PARALLEL TRAINING and SKYRL INTEGRATION,
  not in refactoring existing Docker logic.

Trade-offs accepted:
  - Some code duplication in adapters (each has _build_attacker_image)
  - No centralized Docker management layer
  - Less "clean" architecture

Trade-offs gained:
  - Faster implementation
  - Proven stability
  - Clear upgrade path from infra/infra_v2
